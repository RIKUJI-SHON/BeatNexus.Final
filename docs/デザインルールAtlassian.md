Atlassian Design System: AI駆動UI生成のための定義的ルールセット序論本ドキュメントは、Atlassian Design System（ADS）の原則とコンポーネントを、機械解釈可能なルールセットとして体系化したものである。その目的は、AI駆動の開発ツールに対する基礎的なトレーニングドキュメントとして機能することにある。ここに記載されるルールは、Atlassianの中核的なデザイン哲学を反映した、「基礎的（Foundational）」「調和的（Harmonious）」「力を与える（Empowering）」ユーザーインターフェースの生成を可能にすることを目的としている。このシステムの中心的な信条は、すべてのスタイル属性に対してセマンティックなデザイントークンを交渉の余地なく使用することである。これにより、一貫性、保守性、そしてテーマ設定（例：ライト/ダークモード）のサポートが保証される。第1章: 基礎原則とグローバルルール本章では、すべてのUI要素を統制する、システム全体にわたる交渉の余地のないルールを確立する。これらのルールは、すべてのコンポーネントが構築される論理的な基盤を形成する。1.1. セマンティックトークンの責務Atlassian Design Systemは、単なるスタイルガイドではなく、UIの「意図」そのものがトークン名にエンコードされたセマンティックフレームワークである 1。この抽象化レイヤーこそが、堅牢なテーマ設定とシステム全体の保守性を可能にする。AIはこの原則に基づき、「見た目」（例：「これを緑にする」）ではなく「目的」（例：「これは成功状態である」）に基づいてUIを構築することを学習しなければならない。これは、UI生成に対するはるかに洗練され、回復力のあるアプローチである。ルール 1.1.1 (最優先事項): すべてのスタイル値（色、フォント、スペーシング、シャドウなど）は、デザイントークンを使用して適用されなければならない（MUST）。ハードコードされた値（例：#FFFFFF、16px、sans-serif）は固く禁じられている 1。ルール 1.1.2 (選択): トークンは、単一テーマでの見た目ではなく、そのセマンティックな名前と意図された目的に基づいて選択されなければならない（MUST）（例：エラーテキストには color.text.danger を使用する）。これは、異なるテーマ間でUIの完全性を保証するために不可欠である 1。1.2. コアタイポグラフィルールADSにおけるタイポグラフィ、スペーシング、グリッドは独立したシステムではない。タイポグラフィのルール（4の倍数への丸め）は、8pxを基本単位とするスペーシングシステムのスケール（4pxの増分を含む）と直接対応している 3。この数学的な相互接続性は、システム全体の「視覚的リズム」を生み出す 4。AIは、見出しと段落の間の垂直方向のスペースを定義する際、垂直リズムを維持するために、見出しの行の高さに比例した space.* トークンを選択すべきであるというルールを理解する必要がある。これにより、視覚的に不協和なレイアウトの生成が防止される。ルール 1.2.1 (書体の適用): <code> 要素または Code コンポーネント内のテキストを除くすべてのUIテキストは、Atlassian Sans フォントファミリーを使用しなければならない（MUST）。コードを表すテキストは、Atlassian Mono フォントファミリーを使用しなければならない（MUST） 3。ルール 1.2.2 (スケールとリズム): フォントサイズと行の高さは、計算されたスケールに従わなければならない（MUST）。任意の基本フォントサイズを $S$ とすると、次の大きいサイズは $round\_to\_nearest\_4(S \times 1.2)$ である。サイズ $H$ の見出しの行の高さは $round\_to\_nearest\_4(H \times 1.2)$ であり、サイズ $B$ の本文テキストの行の高さは $round\_to\_nearest\_4(B \times 1.5)$ である 3。ルール 1.2.3 (セマンティックな階層): 見出し要素（<h1> から <h6>）は、ページの論理構造を表現するために使用されなければならない（MUST）。ページには正確に一つの <h1> を含まなければならない（MUST）。見出しレベルをスキップしてはならない（MUST NOT）（例：<h1> の直後に <h3> を配置することはできない） 5。ルール 1.2.4 (使用法): 見出しはコンテンツセクションを導入するために使用する。本文テキストは段落やコンポーネントのコンテンツ（例：ボタンラベル、入力テキスト）に使用する。コンポーネント内で強調を作成するためには、見出しスタイルではなく、より重いフォントウェイトの本文テキストを使用しなければならない（MUST） 6。1.3. システマティックなスペーシングとレイアウトグリッドルール 1.3.1 (トークン化されたスペーシング): すべてのマージン、パディング、要素間のギャップは space.* トークンを使用しなければならない（MUST） 4。ルール 1.3.2 (近接性とグルーピング):強い意味的関係を持つ要素（例：ラベルとその入力フィールド、アイコンとそのテキスト）をグループ化するには、「small」範囲（space.025 から space.100）のスペーシングトークンを使用しなければならない（MUST） 4。異なるUIコンポーネントやセクションを分離するには、「medium」（space.150 から space.300）または「large」（space.400 から space.1000）範囲のスペーシングトークンを使用しなければならない（MUST） 4。ルール 1.3.3 (グリッドタイプの選択):画面の表示領域を最大化することが重要な情報密度の高いページ（例：ダッシュボード、カンバンボード）では、fluid グリッドを使用しなければならない（MUST） 7。行の長さを制限して可読性を向上させるコンテンツ中心のページ（例：記事、ブログ、高密度のデータテーブル）では、fixed グリッド（narrowまたはwide）を使用しなければならない（MUST） 7。ルール ユニバーサルアクセシビリティの責務コンポーネント固有のルール（例：ボタンを無効にしない、モーダルの詳細なフォーカス管理）は、アクセシビリティが単なる「追加機能」ではなく、コンポーネントの設計と振る舞いの根本的な推進力であることを示している 12。送信ボタンが無効にされない理由は、それがタブオーダーから削除され、アクセス不能になるからである。AIは、これらがオプションの「アクセシビリティ機能」ではなく、中核的な構造ルールであることを学習しなければならない。ルール 1.5.1 (コントラスト): テキスト要素とその背景との間の色のコントラストは、少なくとも4.5:1でなければならない（MUST）。大規模なテキスト（24px以上または18px以上の太字）および意味のあるグラフィックの場合、コントラストは少なくとも3:1でなければならない（MUST） 10。ルール 1.5.2 (セマンティックHTML): ページの構造は、セマンティックHTML要素（<nav>、<main>、<section>、<h1>-<h6>など）を使用して定義されなければならない（MUST）。汎用的な<div>および<span>要素は、セマンティックな代替手段が存在しない場合にのみ、スタイリング目的で使用されるべきである（SHOULD） 15。ルール 1.5.3 (キーボードナビゲーション): すべてのインタラクティブな要素は、キーボードを介してフォーカス可能かつ操作可能でなければならない（MUST）。現在フォーカスされている要素には、可視のフォーカスインジケータ（例：Focus ring コンポーネント）が存在しなければならない（MUST） 16。ルール 1.5.4 (テキスト代替): 情報を伝達するすべての非テキストコンテンツ（例：アイコン、画像）は、テキスト代替（例：alt テキスト、aria-label）を持たなければならない（MUST） 15。第2章: コンポーネント固有のルールセット本章では、個々のUIコンポーネントの具体的な構築、外観、および振る舞いのルールを定義する。2.1. フォームと入力コンポーネント2.1.1. フォーム (<Form>)ルール 2.1.1.1 (構造): フォームは、明確なタイトル（<h2> または <h3>）、オプションの説明、および FormSection コンポーネントの下に独自のヘッドラインを持つ論理的にグループ化されたフィールドで構成されなければならない（MUST） 14。ルール 2.1.1.2 (必須フィールド): 送信に必須のフィールドは、そのラベルの隣に RequiredAsterisk コンポーネントで視覚的に示されなければならない（MUST） 14。ルール 2.1.1.3 (ボタンの配置): プライマリ送信ボタンは appearance="primary" を持つ Button でなければならない（MUST）。キャンセルアクションは Link または appearance="subtle" を持つ Button であるべきである（SHOULD）。単一ページのフォームでは、ボタンは左揃えで、プライマリボタンが最初（最も左）に配置される 14。ルール 2.1.1.4 (送信): フォームが無効であっても、プライマリ送信ボタンを無効にしてはならない（MUST NOT）。代わりに、バリデーションフィードバックはエラーメッセージを通じて提供されなければならない（MUST） 12。2.1.2. テキストフィールド (<Textfield>)ルール 2.1.2.1 (ラベリング): すべての Textfield は、その htmlFor 属性がテキストフィールドの入力要素の id と一致する、対応する <Label> コンポーネントを持たなければならない（MUST） 18。ルール 2.1.2.2 (ヘルパーテキストとエラーテキスト): 情報提供のガイダンスは HelperMessage コンポーネントを介して提供されるべきである（SHOULD）。バリデーションフィードバックは ErrorMessage コンポーネントを介して提供されなければならない（MUST）。これらのメッセージは入力フィールドの直下に表示される 17。ルール 2.1.2.3 (バリデーション): エラーメッセージは、何が問題で、どのように修正すればよいかを明確に述べなければならない（MUST） 20。肯定的なフィードバック（例：「ユーザー名が利用可能です」）を提供する場合を除き、ユーザーが入力中にリアルタイムで表示されるのではなく、フォーカスを失った時（onBlur）またはフォーム送信時に表示されるべきである 14。2.1.3. ボタン (<Button>)ルール 2.1.3.1 (外観のロジック):ビュー（例：ページ、モーダル）には、appearance="primary" を持つボタンを最大1つまで含めることができる（MUST）。これは主要なコールトゥアクションのために予約されている 12。破壊的で元に戻せないアクション（例：「削除」、「削除」）は、appearance="danger" を使用しなければならない（MUST） 12。他のすべての標準的なアクションは、デフォルトの外観を使用しなければならない（MUST） 21。appearance="warning" は、重大で望ましくない可能性のある変更に先立つアクションに使用されるべきである 21。ルール 2.1.3.2 (無効状態): ボタンは無効にされるべきではない（SHOULD NOT）。アクションが利用できない場合、ボタンはアクティブなままであるべきである。クリックすると、なぜアクションが実行できないか、そしてそれを有効にするために何が必要かを説明するメッセージ（例：Inline message、Flag）が表示されなければならない（MUST） 12。2.2. メッセージングコンポーネント2.2.1. コンポーネント選択ロジック（決定木）ルール 2.2.1.1: メッセージを表示するには、以下の条件の階層に基づいてコンポーネントを選択する。もしメッセージがシステム全体にわたり、重大で、潜在的なデータ損失または機能損失に関連する場合（例：「サーバーへの接続が失われました」）、Banner を使用しなければならない（MUST） 22。そうでなく、もしメッセージがユーザーのアクションへの応答であり、永続する必要のない一時的な確認やアラートを提供する場合（例：「アイテムをゴミ箱に移動しました」）、Flag を使用しなければならない（MUST） 22。そうでなく、もしメッセージがUIの特定の可視セクションの状態に関するものである場合（例：「これらの検索結果は不完全です」）、そのセクション内で Section message を使用しなければならない（MUST） 22。そうでなく、もしメッセージが単一のインタラクティブ要素（例：入力フィールド）にコンテキストを提供する場合、Inline message を使用しなければならない（MUST） 25。2.2.2. モーダルダイアログ (<ModalDialog>)ルール 2.2.2.1 (使用コンテキスト): Modal Dialog は、自己完結型のタスク（例：単純なフォーム、確認）を完了するためにメインUIをブロックする場合にのみ使用しなければならない（MUST）。Flag や Section message で表示できる非インタラクティブな情報を表示するために使用してはならない（MUST NOT） 23。ルール 2.2.2.2 (外観): 重大な結果を伴うが必ずしも破壊的ではないアクション（例：「ページを別のスペースに移動」）を確認するには appearance="warning" を使用する。破壊的で元に戻せないアクション（例：「このページを完全に削除」）を確認するには appearance="danger" を使用する 13。ルール 2.2.2.3 (アクセシビリティ: フォーカストラップ):モーダルが開かれると、フォーカスはモーダル内に移動しなければならない（MUST）。初期フォーカスターゲットは、最初のフォーカス可能な要素でなければならず、それはヘッダーの閉じるボタンであるべきである（SHOULD） 13。モーダルが開いている間、フォーカスはその中にトラップされなければならない（MUST）。最後のフォーカス可能な要素からタブを押すと、最初の要素にループバックしなければならない。最初のフォーカス可能な要素からShift+Tabを押すと、最後の要素にループしなければならない 28。下にあるページコンテンツは不活性（非インタラクティブ）でなければならない（MUST）。モーダルが閉じられると、フォーカスはそれを開くきっかけとなった要素に戻されなければならない（MUST） 28。ルール 2.2.2.4 (閉じる方法): モーダルは、少なくとも3つの方法で閉じることができなければならない（MUST）：Escape キーを押す、専用の閉じるボタン（ヘッダーの X またはフッターの「キャンセル」）をクリックする、そしてオーバーレイの Blanket をクリックする 26。2.3. テキストとデータ表示コンポーネント2.3.1. ダイナミックテーブル (<DynamicTable>)ルール 2.3.1.1 (使用法): データセットがクライアントサイドでのソート、ページネーション、またはドラッグアンドドロップによる並べ替えを必要とする場合、DynamicTable コンポーネントを使用しなければならない（MUST）。静的で非インタラクティブなデータには、より単純な Table コンポーネントまたはネイティブの <table> 要素を使用すべきである（SHOULD） 31。ルール 2.3.1.2 (状態表示):データが取得されている間、テーブルは isLoading={true} を設定しなければならず（MUST）、これによりスピナーが表示され、コンテンツの不透明度が低下する 31。データセットが空の場合、テーブルは emptyView プロパティを介して、なぜテーブルが空であるかとユーザーが取ることができるアクションを説明する、記述的な Empty state コンポーネントを表示しなければならない（MUST） 31。ルール 2.3.1.3 (アクセシビリティ): 列には明確で単純なラベルを付けなければならない（MUST）。複雑なテーブルには、スクリーンリーダーのユーザーにコンテキストを提供するために、記述的な label プロパティが提供されなければならない（MUST） 32。第3章: UIテキスト生成のためのコンテンツとボイスに関する指示本章では、Atlassianのコンテンツ哲学を、UIコピーを生成するためのプログラム可能なルールに変換する。3.1. コアボイス原則のロジック化ルール 3.1.1 (Bold): UIテキストは直接的で自信に満ちたものでなければならない（MUST）。能動態を使用する。アクションには命令形の動詞で文を始める（例：「ページを作成する」、非推奨：「ページを作成できます」）。「多分」「おそらく」のような曖昧な言葉を避ける 33。ルール 3.1.2 (Optimistic): メッセージはユーザーの成功と可能性を中心に構成する。空の状態では、何が欠けているかではなく、ユーザーが何ができるかに焦点を当てる（例：「最初の課題を作成して始めましょう」、非推奨：「課題がありません」） 33。ルール 3.1.3 (Practical, with a wink):
